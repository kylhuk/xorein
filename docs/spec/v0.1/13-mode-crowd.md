# 13 — Mode: Crowd (Sender-Key Large-Group E2EE)

Crowd mode provides end-to-end encryption for large server groups (>50 members)
using a sender-key scheme with epoch rotation. It is more scalable than Tree
mode but provides weaker post-compromise security (epoch-level, not per-message).

## 1. Overview

Each member of a Crowd-mode group maintains a **sender key** — a symmetric key
used to encrypt all messages they send during an epoch. Sender keys are
distributed to all group members at the start of each epoch. Recipients decrypt
by looking up the sender's key for the current epoch.

## 2. Epoch structure

### 2.1 Epoch parameters

| Parameter | Value |
|-----------|-------|
| Epoch rotation: message limit | 1000 messages |
| Epoch rotation: time limit | 7 days |
| Epoch rotation: membership change | Yes, always |
| Legacy window (accepted old epochs) | ≤2 prior epochs |
| Epoch key length | 32 bytes |

### 2.2 Epoch ID

Each epoch is identified by a 16-byte random `epoch_id` generated by the
server owner (group administrator) at epoch creation. The `epoch_id` is
included in every encrypted message header.

## 3. Key derivation

### 3.1 Epoch root key

At epoch creation, the server owner generates a 32-byte `epoch_root_secret`
(random). All per-member sender keys are derived from this secret.

### 3.2 Sender key derivation

For each member `m` in the group:

```
sender_key_m = HKDF-SHA-256(
    IKM  = epoch_root_secret,
    salt = b"",
    info = "xorein/crowd/v1/sender-key" || peer_id_m,
    L    = 32,
)
```

where `peer_id_m` is the UTF-8 bytes of the member's peer ID string.

### 3.3 Sender key distribution

The server owner distributes sender keys to each member via Seal-encrypted DM:

```json
{
  "kind": "crowd_key_distribution",
  "scope_id": "<server_id>",
  "body": "<base64url(Seal-encrypted SenderKeyPackage)>"
}
```

The `SenderKeyPackage` payload (before Seal encryption) is:

```json
{
  "epoch_id": "<base64url 16B>",
  "epoch_root_secret": "<base64url 32B>",
  "rotation_trigger": "<membership_change|message_limit|time_limit>",
  "issued_at": 1234567890000,
  "expires_at": 1234567890000,
  "member_peer_ids": ["peer_id_1", "peer_id_2", ...]
}
```

The `SenderKeyPackage` MUST be encrypted using the recipient's current Seal
session. If no Seal session exists with a member, a new X3DH exchange is
initiated before distribution.

### 3.4 Epoch chain

Epoch root secrets are chained for continuity:

```
next_epoch_root = HKDF-SHA-256(
    IKM  = current_epoch_root_secret,
    salt = b"",
    info = "xorein/crowd/v1/epoch-root",
    L    = 32,
)
```

This allows administrators to derive epoch `n+1` from epoch `n` without
fresh randomness (used in membership-unchanged rotations). For
membership-change rotations, a fresh random `epoch_root_secret` MUST be used
(no chaining — exclude removed members).

## 4. Message encryption

### 4.1 Encrypting a message

Each member encrypts their own messages using their own sender key:

```
my_sender_key = HKDF-SHA-256(epoch_root_secret, b"", "xorein/crowd/v1/sender-key" || my_peer_id, 32)
nonce = random(12)
aad = crowd_message_header  // see §4.2
ciphertext = ChaCha20-Poly1305.Seal(my_sender_key, nonce, aad, plaintext)
```

### 4.2 Message header

```
[1-byte version = 0x02]
[16-byte epoch_id]
[32-byte sender_peer_id hash (SHA-256 of peer_id UTF-8)]
[4-byte message sequence number in this epoch, big-endian]
[12-byte nonce]
```

Total header size: 65 bytes. Transmitted in plaintext; authenticated via AEAD.

### 4.3 `Delivery` wire format

```json
{
  "kind": "channel_message",
  "scope_id": "<server_id or channel_id>",
  "scope_type": "server",
  "body": "<base64url(ciphertext)>",
  "data": "<base64url(crowd_message_header)>",
  "ciphertext_format": "crowd/v1",
  "signature": "<base64url hybrid sig>"
}
```

## 5. Decryption

1. Parse `epoch_id` from the message header.
2. Look up the cached sender key for the sender's peer_id in the given epoch.
   - If epoch = current epoch: use `my_sender_key_map[sender_peer_id][epoch_id]`.
   - If epoch = one of the 2 legacy epochs: use cached legacy key.
   - If epoch is older than 2 generations: reject with `EPOCH_EXPIRED`.
3. Verify AEAD tag. If invalid: reject with `AEAD_AUTHENTICATION_FAILURE`.
4. Verify hybrid sender signature on `Delivery`.

## 6. Rotation

### 6.1 Rotation triggers

The server owner's client MUST initiate a rotation when:
- A member joins the group.
- A member leaves the group (immediately on leave detection).
- The message counter for the current epoch reaches 1000.
- The current epoch is older than 7 days.

### 6.2 Rotation procedure

```
if rotation_trigger == MEMBERSHIP_CHANGE:
    generate fresh random epoch_root_secret
else:
    epoch_root_secret = next_epoch_chain(current_epoch_root)

distribute SenderKeyPackage to all current members via Seal DMs
increment epoch counter
set epoch_id = random(16)
```

### 6.3 Legacy window enforcement

Nodes MUST retain at most 2 previous epoch sender keys. Messages arriving
with an epoch older than the 2 retained prior epochs MUST be rejected:

```
if msg.epoch_id not in {current_epoch, prev_epoch_1, prev_epoch_2}:
    reject with EPOCH_EXPIRED
```

## 7. Security properties

| Property | Crowd mode |
|----------|-----------|
| Confidentiality | Group E2EE (ChaCha20-Poly1305) |
| Forward secrecy | Per-epoch (not per-message) |
| Post-compromise security | On membership change (forced fresh epoch) |
| Store-now-decrypt-later | Resistant via Seal-encrypted key distribution |
| Relay opacity | MUST be enforced |
| Sender authentication | Yes (hybrid signature on Delivery) |
| Scalability | O(N) key distribution; suitable for >50 members |
